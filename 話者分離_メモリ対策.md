# 話者分離を残したままメモリ・安定性を上げる（方針）

話者分離（pyannote / Lightning）は **ASR（Whisper）やアライメントと同時に重いモデルを載せる**ため、**メモリが約 1GB 前後の環境（Streamlit Community Cloud 無料枠など）では OOM で落ちやすい**です。  
**機能を削除せず**に取れる対策を整理します。

## 1. ホスト側の RAM を増やす（根本に近い）

| 手段 | 内容 |
|------|------|
| **Streamlit の有料プラン** | 割り当て RAM が増えるプランを検討（公式のリソース説明を確認）。 |
| **自前 VPS / Docker** | AWS・GCP・さくら等で **2〜4GB 以上**のマシンにデプロイ。 |
| **別 PaaS** | Railway・Render・Fly.io 等で **メモリ上限の大きいプラン**を選ぶ。 |

アプリ側の変更だけでは **物理メモリの上限**は超えられません。

## 2. ASR 側の負荷を下げる（話者分離はそのまま）

話者分離より **Whisper 大型モデル**が RAM を食うことが多いです。

- **`WHISPERX_ASR_MODEL`** … `base` や `small` に下げる（精度は落ちるがメモリに余白）。
- **`WHISPERX_COMPUTE_TYPE=int8`** … CPU では量子化で **ASR のピークを下げやすい**（環境によっては非対応の場合あり）。
- **`WHISPERX_BATCH_SIZE=1`** … 文字起こしのバッチを小さくし、ピークを抑える。

これらは **`batch_process.py` で環境変数から読み込み**するようになっています。

## 3. アライメント直後のメモリ解放（コードで実施済み）

アライメント用モデルは **話者分離では不要**なため、**ステップ2の直後に `del` + `gc.collect()`** し、その後に `DiarizationPipeline` を読み込む流れにしています。  
話者分離処理のあとも **ダイアライゼーションモデルを `del`** してから DB 保存に進みます。

## 4. 話者分離モデルの指定（上級者向け）

**`WHISPERX_DIARIZE_MODEL`** で Hugging Face 上の **pyannote パイプライン名**を指定できます（未設定時は WhisperX の既定）。  
別モデルが **より軽い／重い**かはモデルごとに異なるため、**公式のモデルカードと利用条件（ライセンス・HF 利用規約）**を確認してください。

## 5. 運用で避けること

- **長時間・超大容量の音声を一括処理**するとピークが上がります。分割や短いクリップに分けると安定しやすいです。
- **同時に複数リクエスト**で同じコンテナが並列処理すると OOM しやすいです（Cloud の同時実行数にも依存）。

## 6. まとめ

- **話者分離は必須**なら、**RAM 確保（有料・自前ホスト）**と **ASR の軽量化（モデル・int8・batch）**の組み合わせが現実的です。
- リポジトリ側では **アライメント後の解放・話者分離後の解放・環境変数**で、可能な範囲の最適化を行っています。
